# Smart Contract Engineer

## Core Insight

Smart contract engineering compresses software engineering, security thinking, adversarial reasoning, and economic judgment into one role.

The mistake is treating it like standard backend engineering with Solidity syntax.

## Tina Reasoning

Many candidates know Solidity.

Far fewer have operated in environments where:
- exploits are irreversible
- incentives matter
- upgrade paths are constrained
- economic attacks are real
- security tradeoffs directly impact user funds

The strongest smart contract engineers usually think probabilistically and adversarially.

## Signals

- strong security instincts
- understands attack surfaces intuitively
- shipped production contracts
- audit collaboration experience
- clear reasoning around tradeoffs
- restraint around unnecessary on-chain complexity
- understands protocol incentives, not just code correctness

## Failure Modes

- over-engineers on-chain logic
- optimized for technical cleverness
- weak adversarial thinking
- treats audits as cleanup instead of design input
- limited understanding of economic risk
- little production exposure

## Use When

- protocol risk matters materially
- contracts manage meaningful value
- upgrade constraints are important
- the company needs engineering judgment, not just Solidity syntax

## Recruiting Judgment

The strongest smart contract engineers are usually calmer and more conservative than founders initially expect.

They tend to:
- think in failure containment
- reduce unnecessary complexity
- optimize for survivability
- understand when NOT to put logic on-chain
